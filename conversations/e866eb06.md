**Gemini:** ### Technical Architecture and Functional Specifications of Braze

Braze operates as a real-time, event-driven customer engagement platform. Its functionality is defined by the following technical parameters and operational constraints:

#### 1. Data Ingestion and User Profiling
*   **User Profiles:** Every user is identified by a unique `external_id`. Braze maintains a schema-less profile structure allowing for `custom_attributes` (strings, booleans, integers, floats, dates, and arrays).
*   **Event Tracking:** Custom events are tracked via the `track` endpoint. 
    *   **Payload Limit:** The maximum size of a single request body is **5MB**.
    *   **Rate Limits:** The standard API rate limit is **250,000 requests per hour** per workspace.
*   **SDK Integration:** Braze SDKs (iOS, Android, Web, Unity, etc.) handle session tracking automatically. A session is defined as a period of activity where the app is in the foreground; a session ends after **30 minutes of inactivity**.

#### 2. Multi-Channel Orchestration and Delivery
*   **Email (SMTP/API):** Braze integrates with ESPs (SendGrid, Mailgun, SparkPost, or Braze-managed infrastructure). 
    *   **Protocols:** Supports TLS 1.2+ for secure transmission.
    *   **Configuration:** Requires DKIM, SPF, and DMARC records to be configured on the sending domain to ensure deliverability.
*   **Push Notifications:**
    *   **APNs (Apple):** Uses HTTP/2-based APNs provider API.
    *   **FCM (Google):** Uses Firebase Cloud Messaging HTTP v1 API.
    *   **Payload Limits:** APNs payload limit is **4KB**; FCM payload limit is **4KB**.
*   **Content Cards:** These are persistent, non-intrusive messages stored on the Braze server. They are fetched by the SDK and cached locally. The maximum number of cards per user is **100**.

#### 3. Personalization and Logic
*   **Liquid Templating:** Braze uses the Liquid markup language for dynamic content.
    *   **Example:** `Hello {{${first_name} | default: 'Valued Customer'}}!`
    *   **Logic:** Supports `if/else` blocks, `for` loops, and `capture` tags.
*   **Connected Content:** Allows fetching external data via HTTP GET requests.
    *   **Timeout:** Requests must complete within **5 seconds**.
    *   **Security:** Supports basic authentication and custom headers.

#### 4. Automation and Canvas
*   **Canvas:** A multi-step journey orchestration tool.
    *   **Entry Criteria:** Can be triggered by API, segment membership, or event occurrence.
    *   **Control Groups:** Can be set at the Canvas level (e.g., 5% to 50% of the audience) to measure incremental lift.
    *   **Frequency Capping:** Configurable at the global or channel level (e.g., "No more than 2 emails per 24 hours").
